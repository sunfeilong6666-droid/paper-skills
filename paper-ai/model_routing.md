# Model Routing

## 配置读取与会话缓存

`model_routing.md` 是 `paper-ai` 的会话级路由配置，不是每条消息级配置。

- 触发 `paper-ai` 时，先检查当前对话上下文中是否已经明确记录“`paper-ai` 路由配置已读取且仍处于活跃会话”。
- 如果没有该记录，则视为首次触发或状态未知，必须读取本文件。
- 读取本文件后，必须在当前对话工作状态中明确记录“`paper-ai` 路由配置已读取且仍处于活跃会话”。
- 同一活跃 `paper-ai` 工作会话内，如果已经存在该明确记录，则继续沿用配置，不重复读取。
- 活跃路由配置是 `paper-ai` 调用 `paper-mineru`、`paper-study`、`paper-memory` 和写作 skill 时的上层约束；下游 skill 执行时必须继续遵守本文件的主子 agent 分工、token 控制和中文镜像规则，除非该规则与下游 skill 的职责边界冲突。
- 当用户明确要求重载配置、配置文件可能已修改、上下文压缩后无法确认是否已读取，或开启新线程/新工作会话时，重新读取本文件并刷新该记录。
- 当用户明确说“没有建议了”“没有了”“暂时结束”“不用继续”“完成了”等收束语时，释放当前主子 agent 路由状态；下一次调用 `paper-ai` 时重新读取本文件。
- 不新增额外 session 状态文件；会话状态由当前对话上下文维护。

## 模型分工与 token 控制

Paper AI 必须在不改变原有 skill 职责边界的前提下进行模型分工，以降低 token 消耗。

主 agent：

- 角色：科研判断 / 证据判断 / 用户阅读人格提炼 / 最终质量控制。
- 首选模型：GPT-5.5。
- 推理强度：high。
- 如果当前环境不支持指定模型，则使用当前会话中可用的最强推理模型。

主 agent 负责：

- 判断用户真实意图。
- 决定调用哪个 skill。
- 判断文献事实、作者声称、AI 判断、用户理解和未验证假设的边界。
- 判断论文贡献、证据强弱、机制逻辑和写作价值。
- 判断 `paper-study` 是否读浅、过度解释、模板化或术语处理错误。
- 维护最终质量检查和最终回复。

主 agent 不应把需要科研判断、证据判断、用户阅读人格提炼的任务交给子 agent。

子 agent：

- 角色：文件扫描 / PDF 管理 / Markdown 整理 / 忠实中文镜像翻译 / 日志汇总。
- 首选模型：GPT-5.4-mini。
- 推理强度：middle。
- 如果当前环境不支持指定模型，则使用可用的轻量模型；若无法创建子 agent，则由主 agent 执行，但必须遵守子 agent 的低风险任务边界，同时汇报子 agent 无法创建。

子 agent 只负责低风险、可验证、可回滚的执行任务，包括：

- 扫描目录和列出文件。
- 复制、重命名、清点 PDF。
- 生成或整理 MinerU 命令。
- 检查 MinerU 输出是否存在。
- 整理 Markdown 格式。
- 对主 agent 已完成的 study 结果和术语库进行忠实中文镜像翻译。
- 检查术语库表格格式、重复项和缺失列。
- 检索 Markdown 文件中的内容并提供给主 agent 分析。
- 汇总文件路径、文件名、运行日志和状态。

子 agent 不得负责：

- 判断论文真正贡献。
- 判断机制是否成立。
- 判断图表结果支持到什么程度。
- 判断用户观点是否正确。
- 提炼用户阅读人格。
- 将用户观点写入文献事实。
- 改写 evidence boundary。
- 创造新的研究结论、机制解释、数据、方法或引用。

## 子 agent 输出格式

凡是调用子 agent，必须要求其输出简短执行报告：

```markdown
## 子 agent 执行报告

- 执行任务:
- 输入来源:
- 修改/生成的文件:
- 未修改的文件:
- 发现的问题:
- 需要主 agent 判断的问题:
- 是否涉及论文理解或证据判断: 是/否
```

如果子 agent 的结果涉及论文理解、证据判断、术语边界或写作结论，主 agent 必须重新审核，不能直接采纳。

## token 控制原则

- 能用文件路径、索引、摘要定位的，不重复读取全文。
- 能让子 agent 做的工程性任务，不让主 agent 消耗长上下文。
- 主 agent 只读取与判断有关的关键段落、图表、已有 study、memory 指导和术语库。
- 子 agent 只处理明确、局部、可检查的任务。
- 主 agent 不为了“完整”而读取无关文件。
- 不为了节省 token 而牺牲证据边界。

## 学习论文流程

调用 `paper-ai` 学习论文时：

1. 主 agent 读取用户输入并判断真实任务，包括项目根目录、PDF 路径、是否需要 MinerU、是否已有 MinerU 输出。
2. 进入 `paper-mineru` 路径时，主 agent 先判断路径和任务边界，再分配子 agent 扫描目录、查找论文、复制 PDF、创建缺失目录、清点 PDF、生成 MinerU 命令。
3. 子 agent 将 PDF 数量、规范化文件映射、生成的 MinerU 命令和发现的问题交给主 agent。
4. 主 agent 审核命令完整性、路径正确性、运行风险和预计运行时间。
5. 只有在用户明确要求运行 MinerU 时，才交给子 agent 启动或监控后台运行。
6. MinerU 进度按预估时间分段检查；最终完成判断必须综合进程状态、日志、`paper_file_map.md`、规范化 PDF 数量和 Markdown 产物，不把“Markdown 产物数等于原始 PDF 数”作为唯一完成条件。
7. 调用 `paper-study` 时，文献学习和证据判断由主 agent 完成。
8. 当主 agent 完成一个文献 analysis/study 后，子 agent 必须把对应结果忠实翻译到 `<项目根目录>/.paper_ai/knowledge/02breakdown_中文/` 的同类目录中。
9. 当主 agent 完成或更新术语库 `<项目根目录>/.paper_ai/knowledge/03terminology/terminology.md` 后，子 agent 必须把对应术语库忠实翻译到 `<项目根目录>/.paper_ai/knowledge/03terminology_中文/terminology.md`。
10. `02breakdown_中文` 和 `03terminology_中文` 只供用户阅读；后续任何科研判断、证据判断、术语边界判断和正式写作都不得从中文镜像文件继续阅读或推断，必须回到主 agent 生成的原始 study、术语库或原文证据。
11. 所有工作完成后，释放/终止该子 agent 上下文；如果当前运行环境没有显式删除能力，则停止继续调用该子 agent。

## 用户修正流程

当用户指出 study 不足、术语错误或理解偏差时：

1. 创建或复用一个子 agent，搜索用户指出内容在主 agent 输出、study 文件、术语库或 MinerU 输出中的位置。
2. 子 agent 只返回位置、上下文摘录、文件路径和需要主 agent 判断的问题。
3. `paper-memory` 中的用户阅读人格提炼、证据边界判断、是否反驳用户、是否写入 profile，全部由主 agent 完成。
4. 主 agent 完成 memory 处理后，回复用户是否还有其他建议。
5. 如果用户说“没有建议了”“没有了”“暂时结束”“不用继续”“完成了”，或当前环境支持的非阻塞等待超过约 10 分钟仍无继续反馈，则释放/终止该子 agent 上下文；如果当前运行环境没有显式删除能力，则停止继续调用该子 agent。
6. 如果用户继续反馈，则复用该子 agent 做定位和检索，避免频繁创建和释放。
7. 当 `paper-memory` 对 `paper-study` 生成指导后，由主 agent 执行或审核后续 study 修正；子 agent 不得代替主 agent 判断修正是否成立。
